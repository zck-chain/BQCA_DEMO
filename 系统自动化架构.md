## 二、 【架构升级】三大核心技术设计

### 1. 动态提示词热插拔编译（Real-time Prompt Engineering）
*   **解决方案（动态 DDL 注入编译）**：
    1.  用户在前端“AI 智脑高级配置面板”输入自定义参数：`temperature = 0.2`，`max_output_tokens = 2048`，以及自定义的专家提取提示词。
    2.  点击分析时，这些参数发给后端，后端在 `sql_templates.py` 中调用模板。
    3.  通过 Python 格式化，生成完全定制化的 SQL，在 BigQuery 瞬间编译热部署生成分析视图 `v_stage2_routed_extractor`。

### 2. 增量大模型解析防重算机制（Incremental Cold/Hot Separation）
*   **痛点**：若每次分析都全量扫描 GCS 上的文件，已解析过的大量历史文件会被重复调用 LLM 分析，造成极高且无意义的 API 调用费用。
*   **冷热隔离自动化流**：
    1.  当用户点击分析时，Python 后台先拉取 BigQuery 物理结果表（或预览缓存表）中已存在的 `uri` 集合。
    2.  Python 在编译 BigQuery 外部对象表读取范围时，在 DDL 中动态拼接增量排除 SQL 过滤层：
        ```sql
        -- 只读取从未被分析过的热数据
        WHERE T.uri NOT IN (SELECT uri FROM `{project_id}.{dataset_id}.t_verified_smart_drive`)
        ```
    3.  **结果**：已确认的历史数据作为“冷数据”不再消费任何大模型 API 配额，只有最新直传上来的文件（热数据）才会触发 Gemini 调用。

### 3. BigQuery 语义 DDL 描述注入（Automated Schema Description Injection）
*   **痛点**：BQCA 关联物理表后，Gemini 智能体必须通过准确翻译 SQL 来回答用户的问题。如果表字段缺乏中英文对照描述，Gemini 在生成 SQL 时容易产生幻觉。
*   **语义层 DDL 自动化注入方案**：
    当用户人工核对完毕，点击一键持久化建表时，后端不仅生成标准的列，还自动触发 BigQuery Schema 的元数据描述注入：
    ```sql
    ALTER TABLE `{project_id}.{dataset_id}.t_verified_smart_drive`
      ALTER COLUMN doc_type SET OPTIONS (description="大模型判定并归类好的文件类别，例如 contract, resume, invoice"),
      ALTER COLUMN parties SET OPTIONS (description="文件中提及的相关主体或公司全称列表（JSON 数组）"),
      ALTER COLUMN amount SET OPTIONS (description="合同发票金额（FLOAT64）"),
      ALTER COLUMN dynamic_attributes SET OPTIONS (description="自适应特有指标 JSON 字典（例如简历的候选人求职岗位和工作年限，合同的延保和交货期等，支持 JSON 穿透查询）");
    ```
    **架构最终效果**：在最底层数据库 Schema 上直接焊死“中文语义钢铁外壳”。当 BQCA 读取元数据进行 NL-to-SQL 翻译时，准确率瞬间拉升至 99% 以上，绝对不胡说八道！
